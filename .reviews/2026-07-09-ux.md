## Prio 1 — Stabilität & Onboarding-Blocker

### 1 · FAB-Speed-Dial: 2 Klicks statt 1 für den häufigsten Use-Case
- **Severity:** P1
- **Wo:** `app/layouts/default.vue:59-81` (FabSpeedDial-Actions-Array)
- **Symptom (User):** Der Mobile-User will eine Ausgabe buchen, tippt auf den blauen `+`-FAB, wählt "Ausgabe" — und landet auf der `/transactions/expenses`-Seite. Dort muss er *erneut* auf "Ausgabe anlegen" tippen, damit der Dialog erscheint. Zwei Klicks, eine Navigation, ein zweiter Klick — wo 1 Klick (FAB → Dialog) gereicht hätte. Schlimmer: Wer *bereits* auf `/transactions/expenses` ist und auf den FAB tippt, bekommt ein No-Op (navigateTo auf aktuelle Route) — der FAB wirkt kaputt.
- **Root Cause:** Die `fabActions` rufen `navigateTo('/transactions/expenses')` statt einen Quick-Add-Dialog zu öffnen. Im Code-Kommentar Zeile 56-58 ist es explizit als "v1: navigieren … als Follow-up Issue" markiert — der Workaround war nie fertig.
- **Empfohlener Fix:** Pages via Query-Param signalisieren, dass der Dialog offen ist (`?new=expense`); der FAB navigiert mit dem entsprechenden Query; jede Page reagiert im `onMounted` auf den Query und öffnet den Create-Dialog automatisch. Alternative: ein globaler `<Teleport>`-Dialog im Layout, der vom FAB direkt geöffnet wird.

### 2 · FAB verdeckt Touch-Targets am unteren Listenende
- **Severity:** P1
- **Wo:** `app/layouts/default.vue:147-165` (FabSpeedDial) + `.content { padding-bottom: calc(5rem + env(safe-area-inset-bottom, 0px)) }` in `app/layouts/default.vue:578`
- **Symptom (User):** Auf Mobile scrollt der User ans Ende einer Transaktionsliste. Der FAB sitzt fix unten-rechts. In den Expenses/Income/Members-Cards (siehe `expenses-mobile-fixed.png`) liegen die Edit- und Delete-Buttons in einer Footer-Row am unteren Rand der Card. Bei mehreren Cards am Bildschirm-Boden *überlappt* der FAB die Edit/Delete-Buttons der letzten sichtbaren Card — vor allem rechts (gleiche horizontale Position). Daumen-Targets kollidieren; der User tappt entweder ins Leere oder auf den falschen Button.
- **Root Cause:** Das `padding-bottom: 5rem` auf `.content` ist die einzige Gegenmaßnahme. Sie schiebt den Listen-Inhalt hoch, lässt aber den FAB in seinem Layer darüber — die *Card-Action-Row* der letzten Card liegt im selben Y-Bereich wie der FAB, weil Cards nicht selbst für FAB-Reserve sorgen.
- **Empfohlener Fix:** Innerhalb der Card-Listen am Listenende ein gleichhohes Padding-Element (z. B. `<div class="list-fab-spacer" style="height: 80px" />`) als letztes Element rendern, oder die Listen-Cards auf Mobile mit einem `padding-right` versehen, sodass Action-Buttons nicht in der FAB-Spalte landen.

### 3 · Login: "**Mock-Modus**" rendert literal mit Sternchen
- **Severity:** P1
- **Wo:** `app/pages/login.vue:120`
- **Symptom (User):** Auf der Login-Seite (die *erste* Seite, die ein Mock-Mode-User sieht) steht im Infoblock: "Da keine Clerk-Konfiguration aktiv ist, läuft die Anwendung im **Mock-Modus**. Du kannst dich…". Die doppelten Sternchen sind **sichtbarer Text**, nicht als Bold-Formatierung interpretiert. Sieht aus wie ein Versehen/Buggy-Markdown. Bestätigt in `.ux-review/shots-after-fixes/login-after.png` (Mobile) und `.ux-review/shots-after-fixes/login-after-desktop.png` (Desktop).
- **Root Cause:** Der String steht in einem normalen `<p>`-Tag. Vue interpretiert keine Markdown-Syntax in Text-Inhalten; die Sternchen landen 1:1 im DOM. Der Fix-3 des alten Reports (Sie→Du) hat den umliegenden Text korrigiert, aber das `**` mitgenommen.
- **Empfohlener Fix:** Entweder die Sternchen entfernen (`"…im Mock-Modus."`) oder den relevanten Teil in `<strong>Mock-Modus</strong>` packen. Letzteres ist konsistenter mit dem visuellen Bruch.

## Prio 2 — Beta-Polish

### 4 · Währung ist freier Text ohne Validierung
- **Severity:** P2
- **Wo:** `app/pages/households/index.vue:176` (Haushalt erstellen) + `app/pages/households/settings.vue:158` (Haushalt bearbeiten)
- **Symptom (User):** Beim Anlegen oder Umbenennen eines Haushalts ist "Währung" ein freies `<InputText>`-Feld mit Placeholder "EUR". Der User kann "USD" eintragen — gut. Er kann aber auch "€", "EUR ", "Euro", "123", "asdf" oder einen leeren String eintragen. `Intl.NumberFormat` mit `currency: "€"` wirft im schlechtesten Fall oder produziert "NaN €" im Format-Output. Folge: Auf dem Dashboard, in Cards und Reports tauchen kaputte Format-Strings auf, und der User merkt erst später, dass die Currency falsch ist.
- **Root Cause:** Beide Forms benutzen `<InputText>` statt einer Select-Komponente. Es gibt keine Whitelist/Validierung im Frontend und (vermutlich) keine im Backend — `formatMoney()` ruft direkt `Intl.NumberFormat` mit dem Wert auf.
- **Empfohlener Fix:** `<InputText>` durch `<Select>` mit den gängigen Codes ersetzen (`EUR, USD, CHF, GBP` — die im CONTEXT.md genannten). Falls freie Eingabe nötig bleibt, beim Speichern serverseitig validieren (3 Großbuchstaben, ISO-4217).

### 5 · Haushalts-Overview zeigt meaningless UUID-Slice als Tag
- **Severity:** P2
- **Wo:** `app/pages/households/index.vue:140` (Aktiv-Card)
- **Symptom (User):** Auf der Haushalts-Übersichtsseite (Mobile: `.ux-review/households-final.png`, After-Fix: `.ux-review/shots-after-fixes/households-mobile-no-eyebrow.png`) steht oben in der "Aktiv"-Card ein grauer, abgerundeter Tag mit dem Inhalt "9bff8d9f". Der User fragt sich: was ist das? Eine ID, ein Fehler-Code, ein geheimer Schlüssel? Es hat offensichtlich keine User-Bedeutung, blockiert aber visuell prominent die Card.
- **Root Cause:** `<Tag :value="currentHousehold.id.slice(0, 8)" severity="secondary" rounded />` — der Code rendert die ersten 8 Zeichen der internen DB-ID als Tag. Das ist ein typisches "Dev-Marker leakt in Production"-Pattern (gleiche Kategorie wie die gefixten MEILENSTEIN-Eyebrows).
- **Empfohlener Fix:** Tag komplett entfernen. Falls eine Identifikation nötig ist, nur im Settings-Dialog als Kopier-fähiger `ID: 9bff8d9f…`-Block mit "ID kopieren"-Button.

### 6 · MEMBER sieht disabled "Einladen"-Button ohne Erklärung warum
- **Severity:** P2
- **Wo:** `app/pages/households/members.vue:190-198` (Einladen-Button) + `app/pages/households/settings.vue:137-142` (zum Vergleich: dort gibt's einen Warn-Hinweis)
- **Symptom (User):** Sarah loggt sich ein, ist MEMBER (kein OWNER), klickt auf "Mitglieder & Einladungen" — der grüne "Einladen"-Button ist ausgegraut, kein Tooltip, kein Hinweistext. Sie fragt sich: ist das kaputt? Habe ich keine Rechte? Wer hat welche? Auf der Settings-Seite wird das richtig gemacht: dort steht "Nur Owner können den Haushalt bearbeiten." — die Members-Seite hat diese eine Zeile nicht.
- **Root Cause:** Im Template fehlt das Conditional-Message-Pendant aus `settings.vue`. Das `canManageHousehold`-Flag wird nur für `:disabled` benutzt, der zweite Anwendungsfall (erklärender Hinweis) wurde schlicht nicht gebaut.
- **Empfohlener Fix:** Direkt unter dem Header einen `<Message v-else severity="info" variant="simple">` einblenden: "Nur Owner können neue Mitglieder einladen. Frag deinen Haushalts-Owner, falls du jemanden hinzufügen willst."

### 7 · Dashboard "Aktiv"-Tag ist tautologisch redundant
- **Severity:** P2
- **Wo:** `app/pages/index.vue:128-145` (Context-Banner)
- **Symptom (User):** Auf dem Dashboard zeigt der obere Banner "Gemeinsamer Haushalt · Währung: EUR · Deine Rolle: OWNER" — und rechts daneben einen grünen Tag "Aktiv". Aber: das Banner erscheint nur, wenn `v-if="activeHousehold"` wahr ist. Es gibt keinen Pfad, auf dem der Banner sichtbar ist und der Haushalt nicht aktiv ist. "Aktiv" ist hier *per Definition* immer true.
- **Root Cause:** Defensiv-Rendering aus der Phase, in der das Banner auch ohne aktiven Haushalt erscheinen konnte. Nach den Fixes für die Empty-States ist der Tag funktionslos.
- **Empfohlener Fix:** Tag entfernen. Oder, falls der Tag einen Zweck haben soll, die Information nutzbar machen — z. B. "Aktiv seit 1. Juli 2026" mit dem Datum, an dem der User den Haushalt erstellt / beigetreten ist.

### 8 · Recurring-Items: monatliches Äquivalent doppelt sichtbar
- **Severity:** P2
- **Wo:** `app/pages/budgeting/recurring.vue:218-247` (Einnahmen-Item) und `262-291` (Fixkosten-Item)
- **Symptom (User):** In der Recurring-Liste zeigt jeder Eintrag das monatliche Äquivalent *zweimal*: einmal als grüner/oranger Tag direkt am Titel ("+ 3.450,00 € / MO" bzw. "− 18,00 € / MO"), und einmal als Aside-Block rechts ("3.450,00 € pro Monatlich"). Für ein 4-Item-Liste ist das visuell redundant und lenkt vom Wesentlichen (Name, Frequenz, Zeitraum) ab. In den Snapshots (`.ux-review/recurring-mobile.png` und `.ux-review/shots-after-fixes/recurring-desktop-no-eyebrow.png`) sieht man's: die Zeile sagt zweimal das gleiche.
- **Root Cause:** `ItemCard` hat den `aside`-Slot für rechtsbündigen Amount; das Recurring-Item nutzt ihn zusätzlich zum monatlichen Tag im Main. Frühere Card-Designs hatten weniger Slots, heute sind beide redundant.
- **Empfohlener Fix:** Eine Variante wählen. Empfehlung: Tag im Main behalten (visuell prägnant, mit Vorzeichen), Aside leer lassen oder entfernen — oder den Aside-Wert umkehren auf den *echten* Periodenbetrag ("3.450,00 € pro Quartalsweise" macht als rechtsbündige Klarstellung Sinn; der Tag im Main zeigt dann das Monatsäquivalent). Aktuell zeigt beides das Monatsäquivalent.

## Prio 3 — Nice-to-have

### 9 · Recurring-Seite: 4 "Neu"-Buttons für 2 Aktionen
- **Severity:** P3
- **Wo:** `app/pages/budgeting/recurring.vue:194-197` (Toolbar) + 215 (Einnahmen-Panel-Action) + 259 (Fixkosten-Panel-Action)
- **Symptom (User):** Auf `/budgeting/recurring` sieht der User auf der Recurring-Seite vier grüne "+ Neu"-Buttons: zwei große in der Page-Toolbar ("Einnahmenplan", "Fixkostenplan"), zwei kleinere in den Panel-Headern ("Neu" beim Einnahmen-Panel, "Neu" beim Fixkosten-Panel). Alle vier lösen jeweils nur 2 verschiedene Aktionen aus. Verwirrend: welcher Button ist "der richtige"? (Alle vier, aber der User muss erst scrollen, um das zu kapieren.)
- **Root Cause:** `<ListPanel #actions>`-Slot wird parallel zur zentralen `<ListPageShell #toolbar>` benutzt. Beide bieten "Neu"-Buttons für die gleiche Domain-Entity.
- **Empfohlener Fix:** Panel-Header-`+ Neu` entfernen, nur die zentrale Toolbar behalten — *oder* die zentrale Toolbar auflösen, wenn das Panel selbst eine klare Aktion hat. Pro Panel genau ein "Neu"-Pfad.

### 10 · Haushalts-Tiles "Mitglieder" und "Einladungen" verlinken auf die gleiche Page
- **Severity:** P3
- **Wo:** `app/pages/households/index.vue:144-151` (drei Tiles unter der Aktiv-Card)
- **Symptom (User):** Auf der Haushalts-Übersicht gibt es drei klickbare Tiles: "Mitglieder 2", "Einladungen 0", "Rolle Owner". Tile 1 und Tile 2 verlinken *beide* auf `/households/members`. Wer auf "Einladungen" tippt, landet auf der Members-Page, muss scrollen bis zum zweiten Panel "Offene Einladungen" — und merkt, dass da "0" steht. Kein Anker, keine Section-Info, kein visuelles Signal "hier sind die Einladungen".
- **Root Cause:** Beide Tiles haben denselben `to="/households/members"` ohne Hash. Die Seite hat keine Anchor-IDs.
- **Empfohlener Fix:** Zweiten Tile auf `/households/members#invitations` linken und im `members.vue` der Section `Offene Einladungen` eine `id="invitations"` geben. Plus: wenn `currentHousehold.invitations.length === 0` ist, sollte der Einladungen-Tile nicht-strong / klickbar sein, weil es nichts zu sehen gibt.

### 11 · Empty-Household-Banner sagt "wähle oben" — auf Mobile ist der Switcher im Hamburger
- **Severity:** P3
- **Wo:** `app/pages/index.vue:149` + `app/pages/households/index.vue:128` + `app/pages/budgeting/recurring.vue` (verwandt: leerer Haushalt)
- **Symptom (User):** Auf Mobile hat der User keinen Haushalt ausgewählt. Das Dashboard zeigt: "Wähle oben einen Haushalt aus, um dein Dashboard zu sehen." Aber "oben" ist auf Mobile nicht der Switcher — der steckt im Hamburger-Drawer (siehe `app/layouts/default.vue:108-180` Sidebar). Der Hinweis in die falsche Richtung.
- **Root Cause:** Desktop-Mentalität: oben = Header. Auf Mobile ist die Sidebar ein Off-Canvas-Drawer, der über den Hamburger-Button oben links erreichbar ist. Der Text wurde nicht responsive gemacht.
- **Empfohlener Fix:** Text responsiv machen, oder generischer formulieren: "Wähle einen Haushalt aus dem Menü, um dein Dashboard zu sehen." Variante: Auf Mobile einen Inline-CTA rendern ("Haushalt auswählen"), der direkt den Drawer öffnet.

### 12 · Owner/Member deutsche Labels im Overview-Tile vs. englische Tags in RoleTag
- **Severity:** P3
- **Wo:** `app/pages/households/index.vue:154` (`Owner` / `Member` als String) + `app/components/RoleTag.vue` (vermutlich `OWNER` / `MEMBER` als Tag)
- **Symptom (User):** Auf der Haushalts-Übersicht zeigt der "Rolle"-Tile "Owner" (deutsche Form, im Fließtext). Auf der Members-Seite zeigt der gleiche User am gleichen Account ein grünes `OWNER`-Tag (englisch, alle Caps). Inkonsistenz — der User fragt sich kurz: ist das das gleiche?
- **Root Cause:** Die Strings wurden an verschiedenen Stellen manuell geschrieben, kein zentraler i18n-Layer. Auch das CONTEXT.md definiert die Rollen als `OWNER` / `MEMBER` (technisch), aber im User-Facing-Text wurde teils lokalisiert, teils nicht.
- **Empfohlener Fix:** Single-Source-of-Truth-Helper `roleLabel(role)` in `app/utils/` (analog zu `frequencyLabel`). Auf der Übersichtsseite: Tile zeigt `roleLabel(role)`. RoleTag rendert weiter `role` als Caps-Tag, weil PrimeVue's `Tag` einen monochromen All-Caps-Stil pflegt. Konsistenz im *Konzept* statt im visuellen Detail.
