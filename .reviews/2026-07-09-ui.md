## Finding 1 — Icon-Buttons ohne `aria-label` (P1 / A11y)

**Wo:** `app/pages/budgeting/budgets.vue:242–251`, `app/pages/budgeting/recurring.vue:236–245, 280–289`, `app/pages/budgeting/savings.vue:224–233`, `app/pages/transactions/expenses.vue:235–244, 281–290`, `app/pages/transactions/income.vue:206–215, 248–257`, `app/pages/households/members.vue:232–241, 273–281, 304–313, 340–348`.

**Symptom:** Sämtliche Edit/Delete/Cancel-Aktionen in den `ItemCard`-Action-Slots und in der `ListTable` rendern `<Button icon="pi pi-trash" ... />` bzw. `pi pi-pen-to-square`, `pi pi-times` — ohne Text und ohne `aria-label`. PrimeVue setzt für `icon`-only Buttons **kein** Default-`aria-label`.

**Impact:** Screen-Reader bekommen „Button" ohne Nutzen; Tastatur-User können die Aktion nicht identifizieren. WCAG 2.1 4.1.2 (Name, Role, Value) verletzt. Betrifft im Worst-Case >30 Aktionen quer durch die App.

**Fix:** `aria-label="Eintrag bearbeiten"` / `aria-label="Eintrag löschen"` als Prop am `Button` setzen — oder eine zentrale Action-Button-Sub-Komponente (`ItemActions.vue`) mit typisierten Action-Objekten, die Label automatisch ableitet.

## Finding 2 — `prefers-reduced-motion` fehlt komplett (P1 / A11y)

**Wo:** Komplette Codebase — `grep -r "prefers-reduced-motion" app/` liefert 0 Treffer. Konkret betroffene Animationen: `app/layouts/default.vue:275,295,312,424` (Sidebar-Slide, Backdrop-Fade), `app/components/BottomSheet.vue:253–278` (Slide-up Transition), `app/components/FabSpeedDial.vue:179–228` (Stagger, Rotation), `app/components/ItemCard.vue:86,141,232,268` (Hover-Transitions), `app/pages/index.vue:340–345` (`@keyframes fadeIn`).

**Symptom:** Alle Animationen laufen ungebremst für User mit reduziertem Motion-Wunsch. WCAG 2.3.3 (Animation from Interactions) verlangt einen Mechanismus zur Deaktivierung nicht-essenzieller Bewegung.

**Impact:** Trigger für vestibuläre Beschwerden, ca. 5–10 % der Nutzer. Besonders auffällig beim FAB-Open (Stagger + Scale + Translate) und der Bottom-Sheet-Transition.

**Fix:** Global in `app/assets/css/base.css` ein `@media (prefers-reduced-motion: reduce) { *, *::before, *::after { animation-duration: 0.01ms !important; transition-duration: 0.01ms !important; } }`-Block; Vue-`<Transition>`-Dauer per JS-Variable `useReducedMotion()` aus dem MatchMedia ablesen und als Prop an `BottomSheet` durchreichen.

## Finding 3 — Sidebar & Header: keine sichtbaren Fokus-States (P1 / A11y)

**Wo:** `app/layouts/default.vue:127–152` (`.nav-item` / `.sub-nav-item` haben nur `:hover` und `.nav-item-active`), `app/components/NavSection.vue:112–125` (Sub-Items gleiches Schema), `app/layouts/default.vue:170–178` (Logout-Button, `outlined severity=secondary`), `app/layouts/default.vue:494–497` (`.toggle-btn` / `.hamburger-btn`).

**Symptom:** Fokus-Ring fehlt oder wird durch das globale `Button`-Reset (PrimeVue Aura) auf einen sehr dezenten Default-Ring reduziert, der auf dem dunklen Hintergrund kaum erkennbar ist. Base.css setzt Fokus-Ringe nur für Form-Fields (Zeilen 180–194) — nicht für Nav-Links/Buttons.

**Impact:** Tastatur-Navigation ist visuell nicht nachvollziehbar. WCAG 2.4.7 (Focus Visible) verletzt. Da die App primär via Sidebar bedient wird, ist das ein Hauptfluss-Problem.

**Fix:** `app/assets/css/base.css` ergänzen um globale `:focus-visible`-Regeln für `.nav-item`, `.sub-nav-item`, `.p-button` (außerhalb der Form-Field-Gruppe) — z. B. `outline: 2px solid var(--color-border-focus); outline-offset: 2px; border-radius: inherit;`.

## Finding 4 — Dashboard (`pages/index.vue`) lebt komplett außerhalb des Design-Systems (P1 / Skalierung)

**Wo:** `app/pages/index.vue` (575 Zeilen, davon ~240 Zeilen CSS ab Zeile 339).

**Symptom:** Die Dashboard-Page benutzt **keinen** der existierenden Building-Blocks (`ListPageShell`, `ListPanel`, `ItemCard`, `Kicker`, `EmptyState`). Stattdessen:
- Ad-hoc-Welcome-Header mit Gradient-Text (Zeilen 122–125, CSS 347–362) — Konflikt mit `ListPageShell` (das eine Page-Hero bereitstellt).
- `.context-banner` (Zeilen 128–145, CSS 364–410) — fast identisch zu `.overview-card` in `households/index.vue` (siehe Finding 8).
- `Card`-Wrapping mit `!important`-Tricks (Zeilen 418–423, 504–509, 18 Vorkommen von `!important` allein in dieser Datei) — Kampf gegen PrimeVue-Card-Styles.
- `.stat-card` + `.stat-icon-wrap` (4 ad-hoc Tones: green/red/blue/purple, Zeilen 412–478) — Token-System hat bereits `--color-accent-success/-danger/-primary/-warning`, aber die Page rollt eigene rgba(34,197,94,0.10)/rgb(34,197,94) etc.
- `.shimmer` (Zeilen 480–492) — eigene Animation, nirgendwo sonst im Projekt verwendet.
- `.budget-track/.budget-fill` (Zeilen 536–549) — dupliziert `ListProgressBar` mit anderen Farben (`rgb(96, 165, 250)` statt `#34d399` etc.).

**Impact:** Jede weitere Dashboard-View reproduziert diesen Wildwuchs. PrimeVue-Update-Bruch-Risiko (die `!important`-Kette koppelt an PrimeVue-Interna). Token-System wird auf der meistbesuchten Seite der App komplett unterlaufen.

**Fix:** Dashboard auf die existierenden Komponenten migrieren:
- Welcome-Bereich → `ListPageShell` mit `eyebrow="Dashboard"` / `title="Hallo, Jan 👋"`.
- Banner → `ListPanel variant="primary"`.
- 4 KPI-Cards → neue `KpiCard`-Komponente (oder `ItemCard density="cozy"` + Tone-Prop).
- Loading → `EmptyState :loading` (existiert bereits).
- Budget-Liste → `ListPanel` + `ItemCard` mit `ListProgressBar` (statt eigenem `budget-track`).
- Shimmer → als CSS-Klasse in `base.css` aufnehmen, damit sie projektweit verfügbar ist.

## Finding 5 — Ad-hoc-Listenzeilen-Stil in 3+ Pages dupliziert (P2 / Konsistenz)

**Wo:** `app/pages/budgeting/budgets.vue:286–345` (`.row-title`, `.row-sub`, `.row-tag`, `.row-tag-muted`, `.amount-secondary`, `.empty-list`), `app/pages/budgeting/recurring.vue:331–387` (identische sechs Klassen + Varianten `.row-tag--green`/`.row-tag--warning`), `app/pages/budgeting/savings.vue:269–320` (wieder identische sechs Klassen).

**Symptom:** Drei Pages definieren byte-identische Style-Blöcke. Die `ItemCard`-Komponente (`app/components/ItemCard.vue:158–216`) bietet über Slots (`#main`, `#meta`, `#aside`) genau die Struktur, die diese Klassen abbilden — und dort ist das Styling bereits zentral. Trotzdem wird `ItemCard` zwar als Wrapper genutzt, der innere `.row-title`/`.row-sub`/`.row-tag`-Stil wird in jeder Page erneut geschrieben. `.row-tag`/`.row-tag--green`/`.row-tag--warning` sind visuell identisch zur PrimeVue-`<Tag>`-Komponente (die schon projektweit genutzt wird, z. B. in `budgets.vue:188–190`).

**Impact:** Änderung am Tag-/Row-Style muss an 4 Stellen synchronisiert werden. Farb-Drift ist garantiert (in `recurring.vue` ist `.row-tag-muted` z. B. nicht definiert, in `budgets.vue` schon — Verhalten unterscheidet sich).

**Fix:** Entweder die inline-Klassen entfernen und PrimeVue `<Tag>`-Komponente (existiert, wird genutzt) verwenden — oder `ItemCard` um eine `#tag`-Slot-API erweitern, die den Style zentral kapselt. Die `.empty-list`-Klasse ist visuell identisch mit `EmptyState` (Loading-Variante); statt eigener Implementation `EmptyState` mit einem `no-data`-Mode.

## Finding 6 — `layouts/default.vue` ignoriert das Token-System (P2 / Konsistenz)

**Wo:** `app/layouts/default.vue:262, 263, 271, 296, 297, 308, 323, 334, 352, 367, 379, 387, 406, 420, 433, 438, 455, 479, 503–507, 511, 525, 538, 555–559, 568` — ~25 hardgecodete Farb-/Background-Werte.

**Symptom:** Das zentrale Token-System in `app/assets/css/tokens.css:21–88` definiert u. a.:
- `--color-bg-page` = `#0b0f19` — das Layout schreibt erneut `background-color: #0b0f19` (Zeilen 262, 568).
- `--color-text-primary` = `#f8fafc` — Layout schreibt `color: #f8fafc` und `color: #f1f5f9` (zwei verschiedene Werte für denselben Job).
- `--color-text-muted` = `#94a3b8` — Layout schreibt `color: #94a3b8` und `color: #64748b` (für „Secondary Text", der im Token-File fehlt).
- Border `rgba(255, 255, 255, 0.05/0.08)` statt `var(--color-border-subtle/-default/-strong)`.
- Brand-Gradient `#3b82f6 → #8b5cf6` ist hardgecodet (Zeile 341) — existiert nirgendwo als Token.
- `rgba(168, 85, 247, ...)` (Env-Tag, Zeile 555–559) ist eine „lila Akzent"-Farbe, die sonst nirgendwo dokumentiert ist.

**Impact:** Das Token-System ist nur halbwertig — Anpassung der Dark-Theme-Farben an einer Stelle greift auf der wichtigsten Layout-Datei nicht. Light-Mode (Aura unterstützt `my-app-light`-Selector) wäre eine Token-Anpassung; aktuell müsste man 25+ Stellen in `default.vue` manuell nachziehen.

**Fix:** Alle Werte in `default.vue` durch `var(--token)` ersetzen. Falls `--color-text-secondary` und `--color-text-subtle` im Token-Set fehlen, dort ergänzen. Den lila Brand-Gradient und die Env-Tag-Farbe als `--color-accent-special` (oder ähnlich) in `tokens.css` aufnehmen.

## Finding 7 — `pages/login.vue` dupliziert `ClerkLoginCard.vue` und rollt eigenes Tailwind-Utility-Set (P2 / Konsistenz)

**Wo:** `app/pages/login.vue:136–328` (193 Zeilen CSS) vs. `app/components/ClerkLoginCard.vue:30–91` (62 Zeilen CSS).

**Symptom:**
1. Die Klassen `.login-card`, `.login-header`, `.logo-container`, `.logo-icon`, `.brand-title`, `.brand-subtitle`, `.clerk-card` sind in beiden Dateien mit **nahezu identischen Properties** definiert (verifiziert: `border-radius`, `background: rgba(30, 41, 59, 0.7)`, `backdrop-filter: blur(16px)`, der Wallet-Icon-Gradient `#3b82f6 → #8b5cf6`, etc.). Die `login.vue`-Variante hat zusätzlich `flex`, `justify-center`, `items-center`, `py-6`, `mb-2/4/6`, `mr-1` als Inline-Utility-Klassen (Zeilen 296–330) — eine Mini-Tailwind-Reimplementierung. PrimeVue liefert PrimeFlex mit, das nicht im Projekt aktiviert ist.
2. Der Login-Button verwendet 9 `!important`-Regeln (Zeilen 276–287) gegen PrimeVue-Defaults.
3. Die Mock-Info-Box zeigt `**Mock-Modus**` als literalen Markdown-Text (Zeile 120) — die `**` werden gerendert, wie in `.ux-review/login-desktop.png` zu sehen.

**Impact:** Visuelle Inkonsistenz zwischen Mock-Login und Clerk-Login ist vorprogrammiert (jede Änderung muss synchron gepflegt werden). 193 Zeilen handgeschriebenes CSS für eine Login-Page ist disproportional.

**Fix:** `login.vue` so umbauen, dass der Mock-Mode-Block die `ClerkLoginCard` (oder ein neu zu extrahierendes `LoginCard`) als reine Slot-Wrapper nutzt — Card-Frame, Header-Logo und Footer-Button in der Card, der Body (Select/Info-Text) als Slot. Die Tailwind-Utility-Klassen durch CSS-Klassen (`.field-stack`, `.field-label`, `.field-help`) ersetzen, die in `base.css` landen. `**Mock-Modus**` aus dem Info-Text entfernen (war ein Copy-Paste-Fehler).

## Finding 8 — Drei „Big-Card"-Styles sind nahezu identisch (P2 / Konsistenz)

**Wo:**
- `app/pages/households/index.vue:183–248` (`.overview-card`, `.overview-card__head`, `.overview-card__description`, `.overview-card__strip`, `.overview-tile`).
- `app/pages/households/settings.vue:165–205` (`.settings-card`, `.settings-row`, `.settings-label`, `.settings-value`).
- `app/pages/index.vue:364–410` (`.context-banner`, `.banner-row`, `.banner-identity`, `.icon-badge`, `.banner-name`, `.banner-meta`).

**Symptom:** Alle drei nutzen denselben Look: 24px border-radius, doppelter Background-Gradient `linear-gradient(180deg, rgba(15, 23, 42, 0.88), rgba(15, 23, 42, 0.66))`, Border `rgba(148, 163, 184, 0.14)`, `box-shadow: 0 18px 40px rgba(2, 6, 23, 0.22)`. Nur Padding und Inner-Slots unterscheiden sich. Insgesamt ~150 Zeilen redundant.

**Impact:** Kein Verhalten-Problem, aber der visuell prägnanteste Container-Typ der App ist nicht abstrahiert. Jede zukünftige „Hero"-Card kopiert das Muster.

**Fix:** Eine `HeroCard`-Komponente (oder Erweiterung von `ListPanel`) mit `variant="hero"`-Mode, die den geteilten Background/Border/Shadow-Block zentral setzt. Drei Klassen-Sets in Pages verschwinden.

## Finding 9 — `households/index.vue` umgeht `EmptyState` mit eigenem Loading-/Empty-Card-Markup (P2 / Konsistenz)

**Wo:** `app/pages/households/index.vue:116–131` und `app/pages/households/index.vue:250–282` (CSS `.loading-state`, `.empty-state-empty`, `.empty-card`).

**Symptom:** Die Seite hat einen eigenen Loading-State mit `ProgressSpinner` (Zeile 120) und eine eigene `.empty-card` für den „Kein Haushalt aktiv"-Zustand (Zeile 124). Die `EmptyState`-Komponente (`app/components/EmptyState.vue`) deckt **genau diese zwei Zustände** ab (`loading` und `noHousehold`) und wird auch in den anderen 7 Pages konsistent genutzt. Die Lade-Visualisierung (Spinner + Card) ist in `EmptyState` aber auch nicht enthalten — es ist ein Kicker-Text, kein Spinner.

**Impact:** Visuelle Inkonsistenz: User sehen auf `households/index` einen Spinner, auf allen anderen Pages einen Kicker-Text-Block. Drei Klassen (`.loading-state`, `.empty-state-empty`, `.empty-card`) definieren dasselbe Card-Frame wie `EmptyState` (`EmptyState.vue:67–79`).

**Fix:** `EmptyState` um `loading-spinner` Slot oder `loading-variant="spinner"`-Prop erweitern. In `households/index.vue` den ganzen `<section v-if=loading>-Block` durch `<EmptyState :loading="loading">` ersetzen. Den Empty-Card-Block ebenfalls auf `<EmptyState :no-household="true">` migrieren.

## Finding 10 — Tag-Severity-Streuung vs. mögliche `SummaryTag`-Kapsel (P3 / Micro-Polish)

**Wo:** 22 `<Tag severity="..." :value="..." />`-Verwendungen quer durch 7 Pages (siehe Grep-Resultat: `budgeting/budgets.vue:188–190`, `budgeting/recurring.vue:189–191`, `budgeting/savings.vue:178–179`, `households/members.vue:185–186`, `households/settings.vue:98–99`, `households/index.vue:98–101, 140`, `transactions/expenses.vue:187–189`, `transactions/income.vue:168–169`, `pages/index.vue:143`).

**Symptom:** Tag wird für zwei völlig unterschiedliche Zwecke genutzt: (a) als Rollen-Indikator (z. B. `OWNER` — `RoleTag.vue` kapselt das sauber) und (b) als Summary-Statistik-Pill in der `ListPageShell`-Header. Für (a) wurde `RoleTag` extrahiert; für (b) ist es eine freie Tag-Instanz. Konsistenz ist okay, aber Severity-Sätze unterscheiden sich (z. B. „Aktiv" Success + rounded, „Mitglieder" Success + nicht-rounded, „Einladungen" Warning + nicht-rounded).

**Impact:** Mikro-Inkonsistenz, kein funktionaler Bruch.

**Fix:** Einen `SummaryTag` (oder `StatPill`) extrahieren, der das Aussehen aus `RoleTag` übernimmt (gerundet, Icon-Slot, einheitliches Padding). Wenn man keinen Wrapper will: Mindestens die `rounded`-Prop und Icon-Slot projektweit vereinheitlichen.

## Finding 11 — Kicker-Töne & Akzent-Farben in Pages teils hardgecodet (P3 / Micro-Polish)

**Wo:**
- `app/pages/budgeting/budgets.vue:308–309`: `background: rgba(59, 130, 246, 0.16); color: #93c5fd;` für `.row-tag`.
- `app/pages/budgeting/recurring.vue:353–369`: `.row-tag` (blau) + `.row-tag--green` `rgba(52, 211, 153, 0.16); color: #34d399` + `.row-tag--warning` `rgba(251, 191, 36, 0.16); color: #fbbf24`.
- `app/pages/budgeting/savings.vue:291–302`: identische Werte wie recurring.
- `app/pages/transactions/expenses.vue:337, 345`: `rgba(59, 130, 246, 0.16); color: #93c5fd;` und `rgba(148, 163, 184, 0.16); color: #94a3b8;`.
- `app/pages/transactions/income.vue:298`: `color: #34d399;` für `.income-amount`.
- `app/components/ListProgressBar.vue:84, 87, 90, 93`: hardgecodetes `#34d399`, `#fbbf24`, `#f87171`.

**Symptom:** Das Token-System hat `--color-accent-primary-text` (`#93c5fd`), `--color-accent-success-text` (`#6ee7b7`), `--color-accent-warning-text` (`#fbbf24`), `--color-accent-danger-text` (`#fca5a5`) in `tokens.css:46–55` — die Pages greifen nicht darauf zu. Stattdessen wird `Kicker` in `Kicker.vue:65–83` sauber über Tokens gesteuert, die `row-tag`/`budget-pill`-Inline-Styles in den Pages aber nicht.

**Impact:** Farb-Drift: `Kicker tone="success"` nutzt `#6ee7b7`, der `.row-tag--green` in `recurring.vue` nutzt `#34d399` — zwei verschiedene Grüntöne für „Erfolg".

**Fix:** Bei Migration von Finding 5 (zentralisieren auf PrimeVue `<Tag>`) verschwindet das automatisch. Falls Eigenbau: Klassen-Sets mit Token-Verweisen neu schreiben.

## Finding 12 — Skeleton- vs. Spinner-Ladezustand: zwei Stile parallel (P3 / Micro-Polish)

**Wo:**
- Skeleton-Shimmer: `app/pages/index.vue:155–158, 480–492` (eigene `.shimmer`/`.shimmer-line`).
- Spinner: `app/pages/households/index.vue:120` (`<ProgressSpinner>`), `app/pages/login.vue:100` (`<ProgressSpinner>`), `app/components/EmptyState.vue` zeigt nur Kicker-Text, keinen Spinner.

**Symptom:** Drei verschiedene Loading-Visuals. Dashboard nutzt Skeleton (gut für Listen-Lade-Zustand), Households und Login nutzen Spinner (gut für Page-Init), EmptyState nutzt Text (am schlechtesten — User denkt evtl. „da passiert nichts"). EmptyState ist zudem die am häufigsten genutzte Loading-Komponente (8 Pages).

**Impact:** User-Erwartung uneinheitlich. EmptyState-Ladezustand wirkt passiv.

**Fix:** EmptyState um `loading-spinner`-Slot erweitern, in dem Default `<ProgressSpinner />` rendert. Skeleton aus `index.vue` als wiederverwendbare Komponente `LoadingSkeleton` (oder CSS-Klasse in `base.css`) extrahieren. Dann projektweit: Listen-Lade = Skeleton, Page-Init = Spinner via EmptyState.

## Was bereits gut gelöst ist (kein Handlungsbedarf)

- **PrimeVue-Collisions sauber behandelt:** `ListTable`/`ListProgressBar`/`FormFieldRow` (Memory im FormFieldRow-Kommentar dokumentiert, Zeilen 11–16). Grep nach `<DataTable`, `<ProgressBar`, `<FormField` ergibt keine Treffer.
- **Token-System für Form-Fields vorbildlich:** `tokens.css:80–88` + `base.css:25–194` lösen das Aura-Default-Inkonsistenz-Problem (Select/DatePicker-Höhe, Focus-Ring, Invalid-State).
- **`RoleTag`-Extraktion** (Zeile 7–14) als Beispiel für saubere Komponenten-Ausgliederung — sollte als Vorbild für `SummaryTag` (Finding 10) dienen.
- **A11y: `aria-label` durchgängig in Layout/BottomSheet/FabSpeedDial** (siehe Grep-Treffer in `default.vue`, `BottomSheet.vue`, `FabSpeedDial.vue`).
- **Touch-Targets:** Layout setzt 2.75rem, Base.css 2.5rem für Form-Fields, ItemCard-Action-Buttons 44pt auf Mobile (Zeile 263) — konsistent WCAG-konform.
- **Mobile-Stacking im ItemCard** (`ItemCard.vue:104–139`) ist sehr sauber gelöst.
- **Mobile-Drawer vs. Desktop-Sidebar** im Layout sauber per Media-Query getrennt.
